# 知识库后端代码重构计划

## 一、重构目标

1. **超高可读性**：类职责单一，命名清晰，代码自解释
2. **职责隔离**：分层明确，调用链路单向流动
3. **调用链路清晰**：同步流程、异步流程边界明确，易于追踪

## 五、分层职责说明

| 层级 | 职责 | 依赖规则 |
|-----|------|---------|
| controller | 接收HTTP请求，参数校验，调用Service | 只能依赖domain.service |
| domain | 业务逻辑编排，事务管理 | 可依赖infrastructure.client和repository |
| infrastructure | 外部服务调用，技术实现细节 | 只能被domain依赖 |
| common | 公共工具，常量，异常 | 可被所有层依赖 |
| config | 配置类，Bean定义 | 可被所有层依赖 |

## 六、重构收益

| 维度 | 当前状态 | 重构后 | 提升 |
|-----|---------|--------|-----|
| 最大类行数 | 342行 | <120行 | 65% |
| 方法平均行数 | 35行 | <20行 | 43% |
| 代码重复率 | 约15% | <5% | 67% |
| 外部依赖耦合 | 直接依赖 | 通过接口隔离 | 完全解耦 |
| 职责清晰度 | 混合多个职责 | 单一职责 | 显著提升 |

## 七、实施建议

### 实施顺序
1. **第一阶段**：创建infrastructure.client包，建立防腐层
2. **第二阶段**：拆分DocumentService，验证新架构
3. **第三阶段**：拆分ChatService，调整其他Service
4. **第四阶段**：整理common和config包
5. **第五阶段**：调整Controller，完成整体迁移

### 注意事项
- 每个阶段完成后进行测试，确保功能正常
- 保持原有功能不变，只调整代码结构
- 及时更新相关配置文件

## 八、风险评估

| 风险点 | 应对措施 |
|-------|---------|
| 功能回归 | 每阶段完成后全面测试 |
| 依赖注入问题 | 使用Spring的@Component和@Autowired确保注入正确 |
| 事务失效 | 确保@Transactional在public方法上，同类调用使用AOP代理 |
